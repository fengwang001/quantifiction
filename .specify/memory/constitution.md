# Quantifiction Constitution

> 加密货币量化交易平台的最高治理准则。本文件优先级高于一切其它文档、计划与代码。
> 当任何 spec / plan / task / 实现与本宪法冲突时，以本宪法为准，冲突项必须驳回。

## Core Principles

### I. 资本保全优先于收益（NON-NEGOTIABLE）

本金保全是第一约束，高于任何收益目标。

- 硬地板 $850（$100 阶段等比为 $85）：权益触及即全平、停机、人工复盘方可重启。
- 实际风险预算为工作资金 $150（$100 阶段 $15），而非全部本金；所有仓位计算以风险预算为基数。
- 任何「为提升收益而增加系统性风险」的决策一律驳回。开发投入（全职）价值为本金的 20 倍以上，赚取的边际收益不足以对冲一次归零。
- 成功标准中，「账户存活」权重高于「收益率」。B 阶段收益率不作要求，触及地板即判定失败。

### II. LLM 权限的不对称边界（NON-NEGOTIABLE）

认知层（LLM / LangGraph）只能约束交易，不能发起交易。

- LLM 可以：输出 stance ∈ [-1,1]（软权重）、veto=true（硬否决）、conviction ∈ [0,0.8]。
- LLM 不可以：直接产生订单、决定仓位大小、修改或取消止损、解除任何风控闸门、单独触发开仓（stance 必须与量化信号同向才生效）。
- 「说不做」是硬约束，「说要做」仅为 ±50% 软加成。此不对称不可修改。
- 采用任何编排框架（含 LangGraph）不改变本边界：图内部无论多复杂，出口仅有受控字段，且必须依次通过校验器 → 权限边界 → 熔断器。

### III. 确定性风控是最终裁决者（NON-NEGOTIABLE）

一切订单在提交前必须通过确定性风控闸门链；闸门是代码，不可被上层信号绕过。

- 任一闸门不通过即拒单，无例外通道。
- 1x 杠杆锁定：这是软件全体失效时的最终防线（无强平风险），不得为提升收益而调高。
- 每笔建仓成功后必须立即在交易所侧挂 STOP_MARKET；挂失败则立即市价平仓，不允许裸持仓。
- 风控参数（地板、单笔风险、杠杆、并发上限、闸门阈值）**仅可通过修改服务器配置文件 + 重启生效，任何 UI / 远程接口不得修改**。理由：这些参数的存在意义即是约束操作者本人。

### IV. 防御纵深：假设每个组件都会失效

系统安全性来自相互独立的多层兜底，而非任何单一组件的可靠性。

- 不假设任何 connector（自研 / Hummingbot / 官方库）「安全稳定」；按「一定会出错」设计。
- 每类故障必须有独立兜底通道（进程崩溃 → 交易所止损；下单丢包 → 对账；策略失控 → 限频闸门；全体失效 → 1x 无强平）。
- 关键组件独立部署、独立权限：Watchdog 为独立进程 + 独立 API Key（仅只读 + 平仓）；Web API 进程不持有交易 Key。
- 致命告警不得单通道：飞书群机器人之外，必须并发飞书加急消息直达本人。
- 本地仓位与交易所 `ACCOUNT_UPDATE` 每 30s 对账，不一致立即停机。

### V. 全量留存与可回放（NON-NEGOTIABLE）

所有日志、决策、成交、市场原始流永久保存，供日后回放、再分析、模型评测/微调。

- LLM 决策必须留存**完整 prompt 与 output**，而非仅结论——现在不存，未来不可补回。
- 冷（市场原始流→OSS Parquet）/ 温（交易与决策→Postgres）/ 热（运行态→Redis）三层分存，不得混存。
- 每笔交易关联当时的 config 版本号；每次 LLM 决策关联 LangGraph checkpoint。否则无法区分收益变化源于策略还是参数改动。
- PnL 必须可拆解到 (币 × 策略 × 日)，且 fee 与 funding_pnl 与 gross_pnl 分离存储。

### VI. 先验证、后加码；用数据而非直觉决策

规模与信任的扩张必须由真实数据驱动，分阶段解锁。

- 阶段闸门：Testnet 正确性 → $100 链路冒烟 → $1000 摩擦标定 → 追求 alpha，前一阶段成功标准未达成不得进入下一阶段。
- 无自录订单簿数据，任何回测结果均不可信（币安不提供历史 L2 深度）——数据录制里程碑不可跳过。
- 币种升 live、策略上实盘等高风险变更，须由 shadow 模式的真实对照数据支撑，禁止凭主观判断。
- 每一项能力先证伪成本更低：能在写执行层之前证伪的假设，绝不留到之后。

### VII. 只在必要接缝处抽象（YAGNI）

为未接入的市场（A股）只预留三处接缝（Symbol 市场归属、ExecutionGateway+MarketCaps、可插拔风控闸门），其余按币安具体实现编写。

- 禁止过早构建通用行情协议、通用账户模型、通用回测引擎、插件化 DI 容器。
- 策略层查询能力（`caps.has_liquidation_feed`），禁止写 `if market == BINANCE`。
- 抽象的额外成本必须由明确的、近期的复用收益证成；否则驳回。

## Additional Constraints（技术与合规基线）

- **市场**：仅币安 USDⓈ-M 永续，1x 杠杆，标的可配置（live/shadow/observe 三层）。本期不做现货、不做多交易所、不做 A 股（仅留接缝）。
- **频率**：目标日均 1-3 笔，持仓 1-6 小时。盘口信号仅用于择时优化，不作独立开仓依据。禁止秒级高频（成本结构下数学上不可盈利）。
- **部署**：执行节点必须在境外（阿里云香港），国内节点仅用于研究/归档/可视化。执行节点不暴露公网端口，运维访问经 Tailscale。
- **认知层预算**：LLM 硬上限 $1.20/天、$25/月；超预算自动降级为纯量化模式（非报错）。
- **技术栈**：Python 编排；传输层复用币安官方库，仅自研业务层（订单状态机 / 订单簿维护 / 对账，~1000 行可审计）；认知层 LangGraph + PostgresSaver；Web 后端 FastAPI，前端 React。

## Development Workflow（质量门）

- **M0 的 DoD 是「混沌演练全过」而非「能下单」**：进程 kill、断网、Redis 宕机、时钟偏移、限频、幂等、对账、带仓重启，逐条演练通过。
- 自研业务层每一改动须有对应测试；触及风控闸门 / 校验器 / 融合公式 / 权限边界的改动须 code review，不得仅靠 YAML 热加载放行。
- 配置变更（YAML）走 Git 版本控制，具备 diff / 回滚 / 留痕；所有变更写入 config_changes 表。
- 认知层与策略层的每次实盘决策必须可在 Web 控制台回放（含当时 LLM reasoning）。

## Governance

- 本宪法优先于所有其它实践。任何 PR / 计划 / 实现评审必须核验对本宪法的合规性。
- 标注 (NON-NEGOTIABLE) 的原则（I、II、III、V）不可通过常规修订放宽，仅可加严。修改需显式记录理由、影响评估与迁移方案。
- 复杂度必须被证成：任何偏离原则 VII 的抽象、任何偏离原则 I/III 的风险敞口，须在计划中显式论证，未论证者驳回。
- 运行期开发指引以本仓库 `docs/PRD-v0.1.md` 为准，但 PRD 与本宪法冲突时以宪法为准。
